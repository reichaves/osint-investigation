# Geolocation Framework

Detailed methodology for determining where a photograph or video was taken, using only the visual content and metadata available.

## When to read this file

Read this when the journalist has uploaded an image, a video, or a link to one, and is asking *where* it was taken. The main `SKILL.md` summarises the loop; this file gives you the substance

## The two-phase model

Geolocation always splits into two phases:

**Phase 1 — Context.** What does metadata and the platform of origin tell you? Cheap to check, low yield in 2026 because most platforms strip identifying metadata, but never skip it.

**Phase 2 — Analysis.** What does the image *itself* tell you? This is where geolocation is actually done. It splits further into foreground analysis (the subject, the object, the action) and background analysis (the environment, the structures, the geography).

## Phase 1: Context — metadata pass

### EXIF — what to extract

When EXIF survives (rarely on social media, more often on direct file shares), look for:

- **DateTime / DateTimeOriginal** — when the camera says it was taken. Camera clocks are often wrong; treat as a hypothesis.
- **GPSLatitude / GPSLongitude / GPSAltitude** — direct coordinates. Useful if present, but easily spoofed.
- **Make / Model** — camera or phone model. Cross-reference with the supposed photographer.
- **Software** — was the image processed (Photoshop, Lightroom, AI tools)? A "Software: Generated by [model]" tag is a strong manipulation signal.
- **Orientation** — sometimes useful to confirm the file has not been re-encoded.

ExifTool is the standard for extraction. In a Python environment: `pip install exifread` or `Pillow`.

### Platform forensics

If the source is a social-media post:

- **Facebook / Instagram / X / TikTok / WhatsApp**: strip GPS and most EXIF on upload. Posting timestamp ≠ capture timestamp. Note timezone — a "posted at 14:23" might be the *server's* time, not the user's.
- **Telegram**: preserves more metadata in "send as file" mode than in "send as photo" mode. If the source is a Telegram channel and you can ask, request the file mode version.
- **Reverse upload check**: if the post claims to be original, run the image through TinEye and Yandex — if the same image appears earlier elsewhere, the post is recycling, not eyewitness.

### Reverse-image search as a first pass

Run the source through, in roughly this order of usefulness for *non-US/EU* content:

1. **Yandex Images** — strongest for European, Asian, Latin American, Middle Eastern content. Especially good with faces, buildings, and "this object somewhere in the world".
2. **Google Lens** — strongest for product identification and US/EU content.
3. **Bing Visual Search** — sometimes surfaces results the others miss.
4. **TinEye** — strongest for finding the *first known appearance* of an image. Useful for distinguishing eyewitness from re-shared.

For a single image, you may need to **crop and search separately**: full-image searches often fail because the engine indexes the whole composition. Crop to a single distinctive element (a sign, a building corner, a specific object) and search that crop on its own.

## Phase 2: Analysis

This is the heart of geolocation. You are pulling information out of the image that the photographer did not intend to communicate.

### Visual clue taxonomy — what to inventory

Make a list of what is *observable* in the image. Do not interpret yet — just list. The goal is to produce as many independent clues as possible, each of which constrains the possible location.

**Linguistic clues**
- Language on signs, banners, posters, license plates
- Currency on price tags
- Phone-number formats (Brazilian +55 11 vs Argentinian +54 11 vs Portuguese +351)
- URL TLDs visible (.com.br, .ar, .pt, .mx)
- Brand names — local-only chains are very strong signals

**Built environment**
- Architecture style (Soviet-era housing blocks, US suburban tract, Latin American colonial, Japanese low-rise, etc.)
- Road markings: lane widths, line colours (Brazil: yellow centre, white edge; UK: white centre, yellow edge), zebra crossings, pedestrian-light style
- Power-line poles: wood vs concrete vs steel; transformer style; insulator type
- Traffic signs: shape and colour standards differ by country (e.g., Brazilian DENATRAN style vs European ECE style)
- Street furniture: bus-stop shelters, mailboxes, fire hydrants are highly country-specific

**Vehicles**
- License-plate format (Brazilian Mercosul: 3 letters + digit + letter + 2 digits, vs old format 3 letters + 4 digits; Argentinian, Mexican, US state plates all distinct)
- Side of the road (drives-on-right vs drives-on-left)
- Vehicle models — some models are sold only in specific markets (e.g., Brazilian-market Volkswagen Gol, Indian-market Tata Nano, Russian-market Lada)

**Natural environment**
- Vegetation (palm species — *Roystonea* in tropics; deciduous broadleaf in temperate zones; conifer types vary by region)
- Topography (mountains, coastline, river bends, lake shapes — match against Google Earth)
- Sky (cloud type and colour can suggest hemisphere or season)

**Solar position — chronolocation**
- The shadow's direction tells you the **azimuth** of the sun (compass bearing). At local noon in the Southern Hemisphere, shadows point roughly south; in the Northern Hemisphere, roughly north.
- The shadow's length relative to the casting object tells you the **elevation** of the sun.
- Combined with a candidate latitude, these constrain the date and time. Use **suncalc.org** or **mooncalc.org** to verify: given a candidate location and a candidate date/time, the predicted shadow should match what is in the image.
- This is called **chronolocation**. It is one of the strongest single tools available, because shadows do not lie (unless the image is composited).

**People**
- Clothing style and weight (heavy coats vs t-shirts → climate / season)
- Uniforms (police, military, school, transport workers — highly country-specific)
- Skin tone, hair, demographics — only as a *probability distribution*, not an identification

### Jurisdictional anchoring

Combine the strongest two or three clues to narrow to **country → region → city**. Examples:

- "Portuguese signage" + "tropical vegetation" + ".com.br domain" → Brazil, almost certainly. Now narrow by accent of signage style (Northeast vs Southeast vs South), architectural era, and topography.
- "Cyrillic signage" + "low-rise concrete blocks" + "yellow taxis with checker pattern" → former-Soviet city, likely Russian-speaking.
- "Spanish signage" + "Andean topography" + "specific bus livery" → Andean South America; further bus-livery research narrows to Lima, Quito, La Paz, etc.

Do **not** stop at "country" — push as far as the clues will go honestly, and stop *before* the point where you would be guessing.

### Map verification

Once you have a candidate area, verify against maps:

- **Google Street View** — the gold standard. Match the camera angle: stand on Street View at the candidate spot, rotate to match the photo's perspective, and check that buildings, signs, and skylines align.
- **Mapillary** and **KartaView** — open-source crowdsourced street imagery. Coverage is patchy but sometimes covers what Google does not.
- **Google Earth (Pro)** — for matching skylines, building footprints, and tall-structure shadows on a known date. Has a *historical imagery* slider, which is essential when verifying when something was photographed.
- **Sentinel Hub EO Browser** — free Sentinel-2 satellite imagery, ~10 m resolution, refreshes every 5 days. Useful for confirming a fire, flood, military movement, or construction visible in the image.
- **Planet Explorer** — daily 3 m imagery, paid (some access via journalism partnerships).
- **Overpass Turbo** — queries OpenStreetMap for unusual combinations. For example, "find every petrol station within 50 m of a railway crossing in São Paulo state" — this kind of query rapidly narrows candidates from a feature that you would otherwise have to scan visually.
- **Bellingcat's Overpass GUI** — a friendlier front-end to Overpass Turbo for non-coders.

### Confidence calibration

This is the rule, restated from the main `SKILL.md`:

| Confidence | Required evidence |
|---|---|
| **High** | Three or more independent converging clues, **plus** at least one map confirmation (Street View match, satellite match, or shadow-position match) |
| **Medium** | Two converging clues, *or* one strong clue (e.g., a Street View match) but unable to triangulate |
| **Low** | A single suggestive clue, or a region-level narrowing without city-level confirmation |
| **Inconclusive** | The available evidence does not allow narrower than country, or even country |

"Inconclusive" is a perfectly valid finding. Saying "the image cannot be located more precisely than 'somewhere in Latin America' from the available evidence" is more useful to the journalist than a fake-precise guess.

## Common failure modes

**Mistaking re-uploads for eyewitness.** If the image was first published years ago and is being passed off as a current event, that is the story. Always check first-known-appearance via TinEye.

**Trusting a single AI "match".** Reverse-image and similar-image AI sometimes return *visually similar* but not *identical* matches. Always click through and verify the underlying source page.

**Believing manipulated EXIF.** EXIF can be edited with a single command. Treat it as one signal, never the deciding factor.

**Over-reading shadows on cloudy images.** Diffuse light produces no clear shadow direction. Note when this is the case and do not invent precision.

**Composite / AI-generated images.** Look for: physically impossible reflections, asymmetric eyeglasses, hands and ears (still hard for image generators in 2026), text that looks "right" at a glance but is gibberish on close reading. If you suspect AI generation, say so explicitly and recommend the journalist run the image through detection tools (Hive, Sensity, or open-source detectors) — but note these are imperfect.

## A worked example

**Source:** A photo allegedly showing a fuel-truck explosion "in Recife, Brazil, this morning."

**Phase 1 — context.** EXIF stripped (came via WhatsApp). The forwarding chain shows the image circulated 8 hours before the journalist received it. TinEye returns a match dated 14 months earlier on a Pakistani news site.

→ The image is **not** from Recife this morning. The story is the recycling, not the explosion. Stop here. Lead sheet states: "Image is recycled from [URL], originally dated [date], showing event in Pakistan. The 'Recife' framing is false."

This is the most important lesson: a 30-second TinEye check often closes the case.
